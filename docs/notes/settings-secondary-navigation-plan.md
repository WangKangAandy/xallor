# 设置二级导航草案（信息架构）

## 1. 目标

- 在不引入复杂树形交互的前提下，降低设置项增长后的定位成本。
- 保持现有视觉风格与交互节奏，避免“文档目录化”带来的学习负担。
- 为后续新增设置项提供稳定分组边界，减少页面平铺扩容导致的回归风险。

## 2. 设计原则

- 仅做 **2 层导航**（一级分组 + 二级任务入口），不引入第 3 层。
- 左侧导航默认保持简洁，**同一时刻仅展开一个一级分组**（accordion）。
- 一级负责“范围”，二级负责“具体任务”。
- 支持搜索命中后自动定位并高亮对应二级项（后续阶段接入）。

## 3. 导航结构草案

### 3.1 一级分组

- 通用
- 外观
- 搜索引擎
- 新标签页
- 组件
- 隐私与安全
- 关于

### 3.2 二级分组（首批建议）

#### 通用

- 语言
- 打开方式
- 启动内容

#### 外观

- 主题
- 壁纸
- 布局与样式

#### 搜索引擎

- 默认搜索引擎
- 引擎管理（预留）

#### 新标签页

- 首页展示项（预留）
- 启动行为（预留）

#### 组件

- 天气
- 快捷建议
- 最近访问

#### 隐私与安全

- 权限说明（预留）
- 数据清理（预留）

#### 关于

- 版本信息
- 更新日志入口（预留）

## 4. 交互规则

### 4.1 左侧行为

- 点击一级：设置为 active，并展开其二级列表。
- 点击二级：主区滚动/切换到对应 section，并保持该二级高亮。
- 若一级无二级：保持当前“点击即进入”行为。

### 4.2 默认展开策略

- 首次打开设置：默认展开“通用”。
- 若带有深链参数（例如 hash/query）：优先展开目标一级并高亮目标二级。
- 关闭设置后不持久化展开状态（避免状态污染），仅持久化业务配置值。

### 4.3 响应式策略

- 桌面端：左侧固定二级展开区域。
- 小屏：保持一级列表，点击后在同列内展开二级，不做多列树形结构。

## 5. 可用性与无障碍

- 键盘导航：方向键在同级移动，Enter/Space 触发展开或选择。
- `aria-expanded` 标记一级展开状态；二级项使用 `aria-current` 标识当前定位。
- 二级项命名应保持“名词化”，避免句子型标签导致扫描负担。

## 6. 实施路线（建议）

### Phase A（低风险）

- 只改左侧导航结构：支持一级展开与二级点击。
- 主区仍沿用现有 section 组织，不做大规模重排。

### Phase B（结构对齐）

- 将主区 section 与二级导航一一对齐（同名锚点）。
- 接入搜索结果与二级导航联动（搜索命中自动展开并高亮）。

### Phase C（体验优化）

- 对预留二级项按产品优先级逐步落地。
- 补齐导航相关回归测试（展开、定位、键盘可达性）。

## 7. 风险与取舍

- 风险：二级项命名不稳定会导致频繁改文案与测试快照。
- 取舍：先做结构，再做文案精修与信息排序，避免一次性大改造成回归面过大。

## 8. 本草案结论

- 建议从“保持原样”升级为“一级 + 二级可展开”，但控制在两层，保持轻量。
- 本草案优先解决“定位效率”和“扩展边界”问题，不追求一次性重做设置系统。
