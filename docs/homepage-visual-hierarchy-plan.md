# 首页三层视觉节奏 + 组件化布局升级计划

## 0) 产品方向与目标

- 产品定位采用「导航页 + 轻桌面系统」：
  - 保留浏览器新标签页的效率主线（搜索优先）
  - 提供可配置组件能力（Widget），但不做完整 Web OS
- 目标：
  - 视觉层级明确：`搜索框 > 核心组件 > 普通入口`
  - 结构可扩展：常用分组 / 常用应用组件化，可自定义
  - 交互可预期：支持固定位置与约束，不再全靠自动补位

## 1) 首页三层视觉节奏（信息架构）

### 第一层：主入口（搜索）

- 只有一个主角：顶部搜索框。
- 设计要求：
  - 宽度更接近主内容区
  - 背景更实、阴影与模糊略增强
  - 上下留白增大，拉开与组件区距离
- 验收：打开页面 1 秒内，第一视觉落点稳定在搜索框。

### 第二层：核心组件锚点

- 保留 1-2 个高权重组件：
  - 天气作为信息锚点（第二主角）
  - 分组/文件夹作为功能锚点（次于天气）
- 要求：二者不再同权，不出现多个“大主卡”并排抢视线。

### 第三层：普通快捷入口

- 单站点入口作为支持信息层，而非主卡层。
- 要求：
  - 尺寸统一、透明度更轻、标题权重更弱
  - 组关系清晰（一排/一块），避免“每个都像中心”

## 2) 组件化策略（Widget 统一模型）

- 将“常用分组”“常用应用”纳入统一组件体系，而非独立散逻辑。
- 统一抽象：`Widget`
  - `AppWidget`：单图标/应用条（1x1、2x1、4x1）
  - `GroupWidget`：分组预览/可展开分组（2x2、4x2）
  - `InfoWidget`：天气、日历、Todo
- 当前已落地的 `gridTile/gridPanel`、`GridDesktopCardSurface` 作为表面层基础，不再新增手写玻璃配方。

## 3) 交互系统升级（关键约束）

> 组件化后，拖拽规则必须升级为「坐标 + 约束」，而不是仅靠自动补位。

- 从“顺序数组”升级到“布局坐标”模型（x/y/w/h）。
- 引入布局约束：
  - `pinned`：位置固定，不被自动重排
  - `floating`：可重排、可自动补位
- 补位策略可配置：
  - 支持 `no-compact`（保留空位）
  - 仅在指定区域 compact（可选）
- 冲突规则明确：
  - 拖入已占位时：交换 / 拒绝 / 挤出（按组件类型区分）
  - `pinned` 默认拒绝被覆盖

## 4) 编辑模式（自定义入口）

- 引入 `edit mode`（编辑态）作为组件化入口：
  - 显示拖拽手柄、删除、resize 控件
  - 非编辑态保持“可用但不扰民”的浏览体验
- 可选触发：
  - 长按进入（后续）
  - 显式“编辑布局”按钮（推荐先做）

## 5) 设计 token 与复用规则

- 视觉 token（已有 + 增补）：
  - `--hero-*`（搜索）
  - `--anchor-*`（核心组件）
  - `--shortcut-*`（普通入口）
  - `--grid-*` / `--grid-merge-*`（网格卡片与合并高亮）
- 复用规则：
  - 卡片表面统一走 `GridDesktopCardSurface`
  - 标题统一走 `GridItemLabel`（可扩展 inside/bottom）
  - 禁止在业务组件中新增手写 `rgba + blur + shadow` 配方

## 6) 分阶段落地顺序

### 阶段 A：视觉层级定标（低风险）

- 完成三层 token 与映射表
- 搜索框主入口增强
- 天气/分组拉开权重
- 普通入口降权

### 阶段 B：组件化首批落地（中风险）

- 常用应用 -> `AppWidget`
- 常用分组 -> `GroupWidget`
- 新增编辑模式基础能力（拖拽/删除/resize）

### 阶段 C：布局约束升级（高价值）

- 引入 `pinned/floating`
- 坐标布局持久化（含迁移）
- 固定位置不被补位，行为可预期

每阶段都执行：`typecheck + test + build`。

## 7) 验收清单

- 视觉主次：搜索 > 天气 > 分组 > 普通站点
- 空页面首次体验：左上首格可见“添加入口”
- 新增首个图标落位：第一行左起第一格（非居中）
- 拖拽行为：固定组件不会被自动补位挤走
- 主题一致性：浅/深主题层级关系保持一致
- 回归：分页、拖拽、重命名、添加图标流程不退化
